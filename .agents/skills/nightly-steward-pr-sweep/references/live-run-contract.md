# Nightly live-run contract

1. Load `SKILL.md` and one Organization-local `references/organization-policy.yaml`. The configured Organization, seat identity, no-touch exclusions, report target and timezone are hard boundaries. One policy file and one run may describe exactly one Organization. A scheduler may invoke multiple Organization policies sequentially, but their gates and outcomes stay independent.
2. Require policy schema `humanandmachine.nightly_steward_pr_sweep.policy.v2`, exactly one `github_organizations[]` entry and exactly one matching `scope.organization_repository_rosters[]` entry. Its `source` must name the authoritative Organization governance roster and its `required_repositories` must be non-empty, unique, canonical `owner/repo` identifiers for that Organization. Reconcile the list against the named governance source, then verify every listed repository with the configured Steward credential. Missing/legacy schema, zero or multiple Organizations/rosters, an empty roster, placeholder values, mismatched Organization rows, omitted governance repositories, or unreadable required repositories are `BLOCKED` before any GitHub mutation; do not inventory only the visible subset or publish a partial report.
3. Require exactly one matching Organization-local entry in `reporting.organization_reports[]`, with exact `github_organization`, `organization_slug`, `steward_seat`, `target` and `repository_db_root` values. Policy v2 never infers report identity or destination from a global seat/root fallback or another Organization policy. Require `reporting.one_append_only_report_per_organization: true`; missing or `false` is `BLOCKED`.
4. Use the live GitHub API for a complete inventory of every open PR in the configured Organization. A dirty branch, conflict, red check, review request or unresolved thread is a work class, not a terminal category.
5. Preserve owner/no-touch and Productionspace routing. Those PRs stay in the Organization-wide inventory and are routed read-only; another owner lane is never permission to omit them. Never mutate billing, permissions, production configuration, deploys or Releases. Never print or store secrets, raw logs or copied client business data.
6. Track `active_unresolved_threads` (unresolved and not outdated) separately from `all_unresolved_threads` (every unresolved thread). Before any merge, re-read the exact head and require all policy gates at that SHA: mergeability, green required checks, independent approval where required, zero **all** unresolved threads and explicit merge authority. An outdated unresolved thread must be proved addressed and explicitly resolved before merge.
7. One colleague may hold separate Steward responsibility in more than one Organization. Keep inventory, authority, evidence, idempotency key and report publication separate per Organization. Never use one Organization's seat, data repo or report as another's target.
8. For each Organization, generate one strict v1 JSON report with ID `steward-<organization-slug>-<steward-seat>-<YYYY-MM-DD>` and idempotency key `<organization-slug>:<steward-seat>:<YYYY-MM-DD>`, using that Organization's explicit report entry. Validate against `mission-control-steward-report.schema.json`; because it uses `additionalProperties: false`, never add fields or enum values not declared by that schema. Set every `inventory.pull_requests[].unresolved_threads` to the **all-unresolved** count, including outdated threads; keep the active-only count in local classification/evidence where the strict schema permits it.
9. Read `repository_db_root` from the matching `reporting.organization_reports[]` entry. Require that checkout to be clean, on `v3`, and fast-forwarded from `origin/v3`. Publish with the standard writer:

   ```bash
   bun "$HOME/.hermes/runtime/steward-reporting/steward-report-cli.bundle.js" \
     --repository-db-root "$REPOSITORY_DB_ROOT" \
     --input "$REPORT_JSON" \
     --organization "$ORGANIZATION" \
     --seat "$STEWARD_SEAT" \
     --schema "$HOME/.hermes/runtime/steward-reporting/mission-control-steward-report.schema.json"
   ```

10. Accept only writer output `status=created` or idempotent `status=existing`, with matching `reportId`, `relativePath` and `readback.idempotency_key`. Commit and push only the returned report path. Re-run the identical input and require `status=existing` plus the same report ID/path as readback proof.
11. Reruns for the same Organization/seat/date are read-only and idempotent. If an existing report differs or the live queue changed after a terminal report, stop with `report_slot_closed`; never mutate GitHub, overwrite the report or create a duplicate for that date.
12. Finish with one concise result per Organization: status, counts, report ID/path/head, merges/fixes/delegations and the smallest remaining human decision. Mission Control is the durable report surface; ClickUp is forbidden.
